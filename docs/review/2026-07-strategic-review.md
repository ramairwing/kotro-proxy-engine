# Kotro Proxy Engine — Strategic Review

*Prepared July 12, 2026. Based on a direct read of the codebase (Go + Rust), CI config, docs, threat model, and GitHub repo state, plus current research on the LLM gateway/proxy market.*

## Executive summary

Kotro is a genuinely well-architected idea two weeks into execution: a local, single-binary proxy that sits between IDE agents (Cursor, Claude Code, Aider) and LLM providers, doing caching, secret redaction, and context compression on streaming SSE traffic. The engineering discipline is unusually high for a project this young — a real threat model doc, tenant isolation with tests, a cancel-storm goroutine-leak audit, dual Go/Rust implementations, k6 load benchmarks. That's rare.

But the project is not yet what its README claims to be. Three things need fixing before "best open source AI tool" is a realistic goal, not a marketing line: the repo has no license file, so it isn't legally open source yet despite saying so everywhere; the flagship "99.3% cost reduction" benchmark is measuring the upstream provider's own prompt caching more than Kotro's; and the feature most central to the product's name — "semantic cache" — is currently exact-match hashing in the shipped Go binary, with real vector similarity stubbed out with a fake embedding function in the unreleased Rust port. None of this is disqualifying, but it means the current README oversells what a user gets today, and that gap will be the first thing a sharp early adopter or HN commenter finds.

The market is also more contested than the README implies. LiteLLM (40k+ stars, MIT) already owns "self-hosted universal proxy." Portkey open-sourced its full gateway under Apache 2.0 in March 2026, complete with real fuzzy semantic caching and guardrails — it now covers most of what Kotro promises, with a much bigger team and community. TokenShift, the one competitor the README names, occupies almost the exact niche Kotro is aiming for (developer-machine-level agent traffic optimization) as a hosted product. Kotro's actual white space is narrower than "premier self-hosted alternative to everything": it's specifically the *local-first, zero-config, single-binary sidecar for coding agents* that never phones home. That's a real, defensible niche — but the project needs to build toward it deliberately rather than accumulating features that duplicate what LiteLLM/Portkey already do well.

## What Kotro actually is today

The repo is two weeks old (first commit June 28, 2026), effectively solo-authored (104 commits from you, 4 from "Cursor Agent"), at v0.3.0 with 11 GitHub releases, and currently sits at 0 stars, 0 forks, 0 watchers on GitHub. That's not a criticism — every project starts there — but it matters for calibrating the review: this is pre-launch, not an established tool losing ground.

Architecturally, there are two parallel implementations: a Go version (Phase 1, "behavioral reference," ~31 source files) and a Rust port (Phase 2, ~25 files) that's positioned as the eventual default. Both implement the same five features: streaming SSE cache, secret redaction, context compression, universal OpenAI/Anthropic protocol translation, and a request circuit breaker for detecting agent infinite-error-loops. The package layout is clean in both languages — cache, guardrail, compressor, proxy, sse as separate concerns — and the test files roughly match the source files 1:1 in Go (26 test files against 31 source files), which is a healthy ratio.

The docs are notably more mature than the code's age would suggest: `docs/security/THREAT-MODEL.md` correctly identifies trust boundaries (client→proxy, proxy→upstream, multi-tenant gateway mode), documents that trust decisions use `RemoteAddr` rather than spoofable `X-Forwarded-For` headers, and calls out that the cache DB contains full response bodies and should be treated as sensitive on shared hosts. `docs/roadmap/90-DAY-ROADMAP.md` is realistic about where the project is (target: "≥100 GitHub stars or equivalent signal" as a phase-3 exit criterion) rather than assuming instant traction. That kind of self-aware documentation is a genuine asset — most solo open-source projects don't have it even at 1.0.

## Feature-by-feature honesty audit

This is the part worth sitting with before doing anything else, because the gap between claimed and actual behavior is the biggest reputational risk to the project.

**Secret redaction** is real and reasonable: a regex pattern set in `internal/guardrail/pattern.go` catches AWS keys, common API key/token formats, DB connection strings, emails, and provider-specific token prefixes (`sk-`, `sk-ant-`), with placeholders restored in the streaming response. It's comparable to what most gateways ship at this stage — not entropy-based or ML-assisted, so it will miss novel secret formats, but it's honestly documented as such in the threat model.

**AST-aware context pruning** (`rust/kotro-proxy/src/compressor/ast.rs`) is real: it uses `tree-sitter` to parse Rust/Python and strip comments and docstrings, with proper span-merging and a size cap to avoid CPU lockup on huge files. This is a legitimately useful, correctly implemented feature.

**The circuit breaker** for agent death-loops is real and sensible: it counts repeated identical cache keys and, past a threshold, injects a synthetic SSE message telling the agent to stop and ask a human, rather than letting it burn API credits in a retry loop. Good idea, clean implementation.

**"Streaming semantic cache"** — the feature the whole positioning rests on — is, in the shipped Go binary, exact-match hashing: `SemanticKey()` is a SHA-256 of the literal system prompt and latest user message, not any notion of semantic similarity. Two prompts that mean the same thing but are phrased differently will not hit the cache. This is a legitimate and useful caching strategy (LiteLLM and most gateways do the same thing), but calling it "semantic" is a naming choice that will read as misleading to anyone who checks, especially set against Portkey and GPTCache, which do real embedding-based fuzzy matching.

**Real vector semantic caching**, added in the Rust port's most recent commits, is currently a stub. The code comment says it directly: *"In a real implementation we would import candle-core and hf-hub to load BERT weights... we will stub the actual ML math and simulate the vector generation."* The `embed()` function generates a fake 384-dimensional vector by summing byte values at modular positions — it has no semantic understanding of the text at all. The cosine-similarity search logic around it (`VectorIndex`, bucket eviction, threshold matching) is correctly engineered, but it's wired to a placeholder encoder. Right now this feature would produce unpredictable false cache hits/misses if enabled, because "similarity" is really just a hash of character bytes.

**"MoE proactive routing"** is a regex keyword matcher (`is_trivial_prompt`) that flags short prompts containing words like "fix typo," "lint," or "format" as routable to a local model instead of the cloud. That's a reasonable heuristic, but calling it "MoE" (Mixture-of-Experts, a specific neural architecture term) borrows credibility from a concept the code doesn't implement. This is the same pattern as "semantic cache" — real, useful engineering wrapped in AI terminology that overstates it.

The throughline: every feature that's *actually shipped and correct* (redaction, AST pruning, circuit breaker, exact-match caching, protocol translation) is solid, unglamorous engineering. Every feature that's *marketed with AI-adjacent buzzwords* (semantic, MoE) is currently either a simpler mechanism wearing a fancier name, or explicitly a stub. For a project trying to earn developer trust, the fix is either to make the vector encoder real before shipping it, or to name things for what they do (`prompt-hash cache`, `keyword router`) until it's real. Developers evaluating infrastructure they'll route API keys through are unusually quick to punish this gap once they read the source — and this is an open-source project, so they will read the source.

## The 99.3% benchmark, read closely

The headline number in the README is worth re-reading carefully: Turn 1 sends 2042 tokens; Turns 2 and 3 are "Local Proxy Miss → DeepSeek Server Hit," billing only 141 and 159 tokens. That's DeepSeek's own upstream prefix caching doing the savings — the README says so explicitly ("Local Proxy Miss"). Kotro's local cache didn't hit at all in this benchmark; the 99.3% reduction is Kotro *forwarding cleanly to an upstream cache that would apply with or without Kotro in the path* (any client hitting DeepSeek directly with a stable prefix gets the same discount). Kotro's real, attributable contribution in that scenario is coordinating the request shape so upstream caching *can* fire (e.g., not needlessly perturbing the prefix) and whatever the local cache saves on top when it does hit. That's still valuable, but it's a materially different, smaller claim than "99.3% cost reduction," and a benchmark-literate reader will notice the difference. If the eval-suite methodology already accounts for this distinction (worth checking `benchmarks/eval-suite/RESULTS.md` before your next README revision), the fix is just presentation: show the two savings sources separately — "Kotro-attributable savings" vs. "upstream prefix-cache savings that Kotro preserves." That reframing is more credible, not less impressive, because it's the number a technical buyer will actually trust.

## The license gap

There is no `LICENSE` file anywhere in the repository, and no SPDX identifier in `go.mod`, `Cargo.toml`, or elsewhere. The README says "Open source — contributions welcome," and GitHub itself shows no detected license. Under default copyright, that means nobody has an actual legal right to use, modify, fork, or redistribute the code — "open source" is aspirational, not true yet. This is the single highest-leverage, lowest-effort fix available: pick a license (MIT or Apache 2.0 are the obvious choices given the competitive set — LiteLLM is MIT, Portkey is now Apache 2.0) and add the file. Every other recommendation in this review is secondary to this one, because right now the project cannot legally be what its README says it is.

## Competitive landscape

| Tool | License / model | Core strength | Where it beats Kotro today |
|---|---|---|---|
| **LiteLLM** | MIT, self-hosted proxy | 100+ provider routing, huge community (40k+ stars, 1,300+ contributors) | Maturity, provider breadth, ecosystem, docs, trust |
| **Portkey** | Apache 2.0 (full gateway, since Mar 2026) | Real fuzzy semantic caching, guardrails (PII, jailbreak/prompt-injection detection), 250+ models, prompt versioning | Real semantic caching (embeddings, not hash), broader guardrails, funded team |
| **Helicone** | MIT (now in maintenance after Mintlify acquisition) | Best-in-class observability UI, per-request cost attribution | Observability depth (opportunity: this lane is now under-invested) |
| **GPTCache** | Apache 2.0, library not proxy | Real embedding-based semantic cache with pluggable vector stores | Actual semantic matching; but it's a library you wire in, no HTTP proxy, no routing/failover — Kotro's proxy form factor is an advantage here |
| **TokenShift** | Hosted/SaaS | Sits at the exact same layer as Kotro — developer-machine coding-agent traffic (Cursor, Claude Code, Copilot, Codex) | Nothing technically; but it's a live, positioned product occupying the niche Kotro wants, and Kotro's own README frames itself in opposition to it |
| **Cloudflare AI Gateway / Kong AI Gateway** | Hosted / enterprise infra | Edge-scale routing, caching, analytics baked into infra developers already run | Distribution — they ship inside platforms teams already use |

Two things stand out. First, Kotro is not filling an empty niche — it's entering a lane with a dominant open-source incumbent (LiteLLM) and a newly-open, well-funded fast-follower (Portkey) that just matched or exceeded Kotro's feature list with real implementations. Second, Kotro's actual differentiation isn't "semantic cache + redaction + compression" as a bundle — Portkey has that bundle too, more maturely. The differentiation is *deployment model*: single static binary, zero external dependencies (no Redis, no vector DB service), local-first with no telemetry leaving the machine unless explicitly configured, and purpose-built for the IDE-agent workflow specifically (VS Code extension, Cursor/Continue/Copilot presets baked into `KOTRO_PROFILE`). LiteLLM and Portkey are both "gateway you run for your org's LLM traffic." Kotro's honest pitch is closer to "the `ripgrep` of LLM proxies for individual coding-agent users" — fast, no-frills, one binary, works offline. That's a smaller but real and defensible claim, and it's more winnable than "premier alternative to everything."

## Recommendations, in priority order

**Fix now (days, not weeks):**

Add an actual `LICENSE` file — MIT is the safer choice for maximum adoption given LiteLLM's precedent, though Apache 2.0 adds patent protection if that matters to you. Rename or caveat the two overstated features: either implement a real embedding model for the vector cache (a small ONNX or candle-based MiniLM model is genuinely tractable and would make "semantic cache" true) or rename it to something accurate until it is; same for "MoE routing" → "keyword-based local routing" or similar, until there's an actual learned classifier. Rework the headline benchmark to separate Kotro-attributable savings from upstream prefix-cache savings — this is a credibility fix, not a marketing downgrade.

**Build next (weeks):**

Ship the real semantic cache. This is Kotro's most differentiated technical claim if it's true, and currently the least true thing in the codebase. A local MiniLM-class embedding model (small enough to bundle or lazy-download, runs fine on CPU for this use case) would make Kotro the only single-binary proxy with *actually local* semantic caching — GPTCache requires a vector DB service, Portkey's is cloud-side. That's a real, ownable claim: "the only local-first proxy with on-device semantic caching, no vector DB, no API calls for the cache itself."

Publish the eval-suite results publicly and keep them current — `benchmarks/eval-suite/RESULTS.md` existing is good; making it a living, re-run-on-every-release artifact (like LiteLLM's and Portkey's own benchmark pages) is what converts skepticism into trust for infrastructure tooling.

Pick one language and stop dual-maintaining. Running Go and Rust implementations of the same five features in parallel is roughly double the maintenance cost for a solo maintainer, and it's not clear either version is meaningfully ahead — the Rust port has the more advanced (if currently fake) semantic cache work, while Go is the "behavioral reference." At solo-maintainer scale, converging on one implementation and archiving the other as a reference/rewrite note will move faster than the current split. If Rust is genuinely the target for distribution reasons (single static binary, no GC pauses under load), commit to it and sunset Go rather than shipping both.

**Grow the project (ongoing):**

Get to a real "first ten users" before optimizing further — the roadmap's own instinct here is right (design partners, not features) and is the correct order of operations: trust artifacts and eval data exist, so the next unlock is a Show HN or similar launch anchored on the corrected benchmark and the real semantic cache, not more features. Consider explicitly *not* competing with LiteLLM/Portkey on breadth (provider count, guardrail catalog) — a comparison table in the README that says "if you need one binary with zero dependencies for your own coding-agent traffic, use Kotro; if you need to gateway traffic for an entire org across 100+ providers, use LiteLLM" is more credible than "premier alternative to everything," and it correctly routes users who'd bounce off Kotro anyway toward a good outcome, which builds trust with the users who do fit.

## Bottom line

The engineering fundamentals here — threat modeling, tenant isolation with tests, leak audits, clean package boundaries — are better than most two-week-old projects, open source or not. The gap to "best open source AI tool" isn't more features; it's closing the space between what the README promises and what the code currently does (license, semantic cache authenticity, benchmark framing), then doubling down on the one differentiator that's real and ownable: a genuinely local, single-binary, zero-dependency proxy purpose-built for coding agents. That's a smaller claim than the current README makes, and a much more winnable one.
