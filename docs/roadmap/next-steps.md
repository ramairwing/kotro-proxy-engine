## P1 — Verify Rust independently, then freeze Go

Revised framing (see conversation log): "match Go's test count" is a proxy metric, not the goal. Go's 74 `#[test]`-equivalents split into two buckets — Go-specific plumbing (`io.Pipe` watchdog behavior, bbolt quirks) that has no meaningful Rust equivalent and shouldn't be ported, and behavioral invariants (tenant isolation, redaction correctness, protocol parsing) that must hold regardless of language and are currently *proven* on Go but only *assumed* to carry over to Rust. The work below targets the second bucket specifically, not the raw count.

- [x] **Redaction correctness — real gap found, fixed, and verified.** Auditing `guardrail/redactor.rs` against `internal/guardrail/pattern.go` found the Rust redactor was missing 6 of Go's 10 secret-detection patterns entirely: password/passwd/pwd fields, and all 4 DB connection string types (postgres, mysql, mongodb/+srv, redis), plus email addresses. This wasn't a test-coverage gap, it was a functional gap — the Rust binary would forward those secret types to upstream providers unredacted. Added the missing patterns and a full test suite (17 tests: one per pattern type, round-trip-through-redaction-map, multi-secret handling, Anthropic system-prompt coverage) where there were previously zero tests for this file. **Verified: 17/17 pass locally** (`cargo test -p kotro-proxy guardrail::redactor`).
- [x] **Tenant/scope isolation — real gap found, fixed, and verified.** `router/scope.rs` had unit tests for `ScopeResolver` in isolation, and `cache/semantic.rs` had unit tests proving different scope strings hash differently — but nothing tested the actual wiring between them (`unified_cache_key` → `scope.key()` → `generate_cache_key`) the way `router/handlers.rs` uses it on a real request. A regression that dropped or hard-coded the scope on the way into cache-key generation would not have been caught by either layer's tests alone. Added 4 tests chaining the full path: different credentials → different cache keys (the core isolation property), same credential → same cache key (isolation without correctness is useless — confirms cache hits still work), missing credentials → shared default scope (documented/intentional), and the trusted-gateway header path isolates tenants the same way. This is the Rust equivalent of Go's `TestCacheIsolation_TenantSeparation`. **Verified: 6/6 pass locally** (`cargo test -p kotro-proxy router::scope`).
- [x] **SSE frame parsing — audited, already solid, no changes needed.** `sse/frame.rs` has 9 existing tests covering fragmentation, Windows line-endings, byte-level round-trips, zero-copy allocation, and two tests explicitly named `*_go_vector` confirming intentional Go-parity test vectors. This was already the best-tested module in the Rust codebase; not a real gap.
- [x] **Cancel-storm leak audit — wired into CI, plus a real bug found in the process.** `benchmarks/run_rust_audit.sh`'s `thread_count()` used macOS-only `ps -M` syntax, which does not work on GitHub's `ubuntu-latest` runners — wiring it into CI as-is would have produced a broken job, not a working gate. Fixed with a `/proc/<pid>/status`-based Linux branch (Go's `run_audit.sh` already used the portable pprof HTTP endpoint, no changes needed there). Added `.github/workflows/cancel-audit.yml`: installs k6, runs `make audit` (both Go and Rust leak audits sequentially, matching the existing local entrypoint), uploads logs on any run. Deliberately a separate, non-blocking workflow rather than folded into the required `ci.yml` — it's a ~1-2 minute k6 load test, not a fast unit-test gate, and this is its first-ever run in GitHub Actions. **Needs a real CI run to confirm** (triggers automatically on the next push to `main`, or via the Actions tab's "Run workflow" button) — promote to a required check on `ci.yml` once proven stable across a few runs.
- [x] Distribution parity: spot-checked npm (`binary-target.js`), Homebrew (`kotro-proxy.rb`), and the curl installer (`install.sh`) — all reference Rust cross-compilation target triples (`aarch64-apple-darwin`, `x86_64-unknown-linux-gnu`), none silently shipping Go. Confirmed, no channel needs fixing.

- [x] Replace the current stub test with real accuracy tests: paraphrase pairs that should hit, unrelated prompts that shouldn't, at a tuned cosine threshold. Done — `semantic_similarity_reflects_paraphrase_vs_unrelated` and `vector_index_lookup_uses_encoder_output` both pass locally (`cb49700` recalibrated one threshold after the first real run showed mean-pooled MiniLM's actual anisotropy baseline; see commit message for the reasoning). All 3 tests in `cache::vector` pass as of the latest local run.
- [x] Benchmark embedding latency overhead and publish it next to cache-hit-rate numbers. Measured locally via `cargo run --release --example bench_embedding` (`rust/kotro-proxy/examples/bench_embedding.rs`, added in `bdf75c5`):

  | Prompt shape | mean | p50 | p95 | p99 |
  |---|---|---|---|---|
  | short (~8 words) | 25.9ms | 25.9ms | 26.6ms | 27.7ms |
  | medium (~46 words, inline code) | 26.7ms | 26.6ms | 27.1ms | 27.5ms |
  | long (~257 words, file-content turn) | 27.3ms | 27.3ms | 27.8ms | 28.2ms |

  One-time model load (warm HF cache, weights already downloaded): 151.8ms — happens once at proxy startup, not per request.

  **Interpretation:** latency is flat across prompt sizes (~26-28ms) — dominated by fixed model compute, not input length, until you approach the 512-token truncation ceiling. In absolute terms this is imperceptible to a human waiting on their IDE, and small relative to typical upstream provider round-trip time (hundreds of ms to seconds). Two things worth being explicit about rather than glossing over: (1) `embed()` runs on *every* request when the vector cache is enabled, including on exact-match and vector-cache *misses* — so this ~26ms is a flat tax paid even when it doesn't produce a hit, not just an amortized cost that comes with a savings payoff. It's ~13x the default `KOTRO_CACHE_HIT_DELAY_MS` (2ms) exact-match replay pacing. (2) ~~`embed()` currently blocks its tokio worker thread synchronously~~ **Fixed and verified** (`d8f3072`): the two call sites in `router/handlers.rs` now go through an `embed_off_thread()` helper that runs the embedding call via `tokio::task::spawn_blocking`. Full local test suite passes (57/57, `cargo test -p kotro-proxy`), including both `cache::vector` tests unaffected as expected.

## P4 — Growth and ecosystem positioning (after P0–P3)

- [ ] Position Kotro explicitly as an **MCP-aware local proxy**, not a generic LLM gateway. The context compressor already touches MCP tool schemas — lean into this in the README/docs and in any launch content, since MCP is the fastest-growing integration surface in the coding-agent space right now and a more specific, timelier claim than "AI proxy."
- [ ] Build a real **extension/plugin surface** so other teams can build on top of Kotro, not just run it. Concretely: a trait-based interface for custom cache backends and custom redaction rules, or a WASM plugin surface for compliance rules; and publish the core logic as a separate library crate (e.g. `kotro-core` on crates.io) that can be embedded, not just invoked as a binary. This is the actual unlock for "company builds their own product on top of Kotro" rather than "company runs the binary."
- [x] **MCP prompt injection scanner + per-session token budget — implemented.** Added two new guardrail modules in `rust/kotro-proxy/src/`:

  - **`guardrail/injection.rs`**: 14 regex patterns covering the primary MCP attack surface (MCPTox 2026: 72.8% baseline attack success rate). Categories: classic override phrases, persona/role hijacking, fake system tags (`[SYSTEM]`, `[[INST]]`, `<COMMAND>`), data exfiltration, agent control injection. Scanner runs before the cache lookup so poisoned requests can never populate the cache or return a poisoned cached response. Mode: warn-by-default (`X-Kotro-Injection-Warning` header) or block (`KOTRO_INJECTION_BLOCK=true`, HTTP 400). 17 unit tests including false-positive regression for `"act as a helpful assistant"` and clean JSON API responses.

  - **`budget/mod.rs`**: Per-scope session token budget tracker backed by `moka::sync::Cache<String, u64>` with 24h idle TTL. 4-chars-per-token estimation. Mode: warn-by-default (`X-Kotro-Budget-Remaining: 0` header) or block (`KOTRO_BUDGET_BLOCK=true`, HTTP 429). Cache hits count as zero tokens (they cost $0). Response headers on every cache-miss request: `X-Kotro-Tokens-Used`, `X-Kotro-Budget-Remaining` (when a limit is configured). 11 unit tests.

  New env vars (all opt-in, no behavior change at defaults): `KOTRO_ENABLE_INJECTION_SCAN`, `KOTRO_INJECTION_BLOCK`, `KOTRO_SESSION_TOKEN_BUDGET`, `KOTRO_BUDGET_BLOCK`. Committed `1f28b94`. **Needs `cargo test` verification on your machine before push.**

- [x] **Supply-chain trust signals — implemented, not yet run.** Added keyless (OIDC) artifact signing via `sigstore/cosign` and an SPDX SBOM to `.github/workflows/release.yml`: `id-token: write` permission added, `anchore/sbom-action@v0` generates `kotro-proxy-engine-<version>.sbom.spdx.json`, and a `cosign sign-blob --yes` loop signs every file already staged in `release-assets/` (binaries, VSIX), producing a `.sig`/`.pem` pair per artifact that the existing `files: release-assets/*` glob in the GitHub Release step picks up automatically — no other job wiring needed. Keyless signing was chosen over a stored key because it proves provenance via the workflow's own GitHub Actions identity against Sigstore's public Fulcio/Rekor infrastructure, with nothing to leak. Documented end-user verification (`cosign verify-blob` with the repo's OIDC issuer/identity) in a new README "Verifying releases" section, including an honest caveat that the `curl | bash`/Homebrew/npm channels don't run this check automatically today. **Not yet verified end-to-end** — this can't be tested locally the way `cargo test` can; needs either a manual `workflow_dispatch` run from the Actions tab or a real `v*` tag push to confirm the SBOM and signing steps succeed and produce valid signatures.
